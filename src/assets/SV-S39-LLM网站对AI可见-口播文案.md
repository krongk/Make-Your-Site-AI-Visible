## 开场：发生了什么

最近有一个变化，很难忽视：很多人遇到问题，不再先去搜索，而是先去问 AI。

你把链接丢给它，它会读一读，然后给出结论。听上去很自然。

但这件事的新闻点，不是“AI 更聪明了”，而是：**你的网站，第一次要面对一种新的读者**。这个读者不看布局，不点导航，也不一定执行脚本。它要的是信息密度。

如果你的网站提供的主要是 HTML，它读到的常常是导航、脚本、样板字。信息有，但噪声更大。

所以本文只做一件事：把“可读性”这件事，按工程的方式做完。

接下来我会按顺序讲 **6 个关键技能**。每讲完一个，我会告诉你下一步讲什么。

---

## 先说第零步：别把门关上

在所有技巧之前，先做一件小事：检查 `robots.txt`。

如果你已经把 GPTBot、ClaudeBot 一类爬虫挡在门外，后面做得再漂亮，也没有入口。

这一段讲完。下面进入第 1 个技能。

---

## 技能 1：`/llms.txt`（给 AI 的站点地图）

第一个技能，是在站点根目录放一个 `llms.txt`。

它的作用很朴素：用 Markdown 列出“你希望 AI 优先读的关键页面”。可以把它当作对话时代的 README。

这里要说一句保留：**没有哪家厂商正式承诺一定读取它**。所以别把它当作“神奇开关”。

更现实的价值在于：当人或工具把 URL 交给模型时，它能快速把模型引到正确的页面上。

这一段讲完。下面是第 2 个技能：让每篇内容都有一个干净的落点。

---

## 技能 2：同址 `.md`（把指针指向干净正文）

第二个技能，是给每个“有内容的 URL”准备一个 Markdown 孪生：同一地址加上 `.md`。

很多人以为这只是格式问题，其实不是。它解决的是“信息密度”的问题。

同一篇文章的 HTML 可能很长，但有效信息只占一部分；把它变成干净 Markdown，模型的上下文压力会小很多。

当然也有代价：如果你同时维护 HTML 和 Markdown，容易漂移。最好的方式，是保持单一事实来源，然后自动生成另一份表示。

这一段讲完。下面是第 3 个技能：如何把 Markdown 的存在告诉各种客户端。

---

## 技能 3：`<link rel="alternate">` + HTTP `Link`（双轨宣告）

第三个技能，是“发现机制”。

一类客户端会解析 HTML 的 `<head>`，你就在里面加一条：`rel="alternate" type="text/markdown"`，指向 `.md`。

另一类客户端根本不解析 DOM，只看响应头。你就再加一个 HTTP `Link` 头，同样指向 `.md`。

它们做的是同一件事：告诉对方“同一内容还有另一种更适合机器读取的表示”。

这一段讲完。下一段讲一个更现实的场景：用户直接把链接贴进 ChatGPT，会发生什么。

---

## 技能 4：视觉隐藏提示（给“贴链接”场景的路标）

第四个技能，解决的是“人把链接贴进对话框”这个场景。

这时模型读到的是渲染后的页面文本，并不一定会去看你的 HTTP 头。

你可以在页面里放一段视觉隐藏的提示，告诉它“这页有 Markdown 版，地址在这里”。再配上 `aria-hidden`，避免读屏器把这段当作正常内容读出来。

它不是必须的，但成本很低，而且对“对话式使用”很直接。

这一段讲完。下一段讲第 5 个技能：把整站内容变成一次性可读的版本。

---

## 技能 5：`/llms-full.txt`（一次性给出更完整的正文）

第五个技能，是 `llms-full.txt`。

它和 `llms.txt` 的关系，类似“目录”和“全集”。前者给你指路，后者把更多内容一次性打包给你。

它更适合文档站，因为文档常常需要跨页引用。博客或营销站要量力而行，必要时重定向到一个更合适的落点也可以。

这一段讲完。最后一个技能，是最“协议化”的做法：内容协商。

---

## 技能 6：`Accept: text/markdown`（用 HTTP 协议解决问题）

第六个技能，是 HTTP 内容协商。

客户端在请求里写 `Accept: text/markdown`，服务器就优先返回 Markdown；否则返回 HTML，并用 `Vary: Accept` 让缓存区分开。

我不是协议专家，但这条路线听上去更稳：它依赖的是 HTTP，而不是某个新发明的“AI 专用 meta”。

这里也要把边界说清楚：不要按 User-Agent 偷偷换正文，那是另一回事，容易被理解成 cloaking。内容协商的前提是：同一信息，只是不同表示。

六个技能讲完了。最后补两段：如何验证效果，以及哪些坑可以少踩。

---

## 如何验证：不测就不知道

不要问“是不是会被 AI 收录”，那是别人系统里的事。

你能做的是记录：`.md`、`/llms.txt`、`/llms-full.txt` 这些端点，谁来访问了。看 `User-Agent`，看 `Referer`，看是否来自 `chatgpt.com`、`claude.ai`、`perplexity.ai`。

注意：前端统计脚本对很多爬虫无效，最好看服务端或边缘日志。

---

## 结尾：更现实的做法

很多“AI SEO 小技巧”看上去很热闹：特殊 meta、注释里藏指令、各种 `ai.txt` 的提案。

它们的问题在于：很难验证、很难统一、也不一定有人读。更现实的做法，是把正文写清楚，把落点做干净，把协议接上，然后用日志验证。

这可能不是最刺激的路线，但通常是最省事的路线。
